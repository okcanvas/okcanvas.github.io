# Expert Record Style Guide

## 핵심 기준

이 문서는 자신을 크게 보이게 하는 소개문이 아니라, 운영 가능한 시스템을 만들기 위해 어떤 제약을 받아들이고 어떤 책임 경계를 세웠는지 남기는 기술 기록이다.

## 사용하지 않을 표현

- 직급명, 최고 수준 표현, 규모 수치를 앞세우는 자기 수식
- 첫 화면에서 고객사 수치를 강조하는 표현
- 어려움과 배움을 막연하게 말하는 회고형 문장
- 좋다/나쁘다 중심의 도구 평가
- 모든 것을 해결했다는 식의 과장된 성과 표현

## 사용할 표현

- 운영 압력: 고객사별 설정, 배포 버전, 장애 이벤트, 외부 연동 실패
- 책임 경계: application, gateway, worker, DB, cache, media server, Agent
- 검증 흔적: audit, trace, hash, diff, rollback, DLQ, revision, sequence gap
- 버린 선택: 더 단순한 방식, 더 빠른 방식, 상용 솔루션, 자유 스크립트의 한계
- 운영 결과: 사람의 기억에서 시스템 기록으로 이동한 지점

## 문장 기준

좋지 않은 문장:

```md
대규모 고객사를 안정적으로 운영했다.
```

좋은 문장:

```md
고객사별 설정, 배포 버전, 업무 도메인, 장애 이벤트가 분리되어 관리되지 않으면 장애 원인 추적이 사람의 기억에 의존하게 된다. 이를 줄이기 위해 이벤트 수집 기준을 고객사·도메인·버전·설정 변경 이력 단위로 나누었다.
```

## 문서 구조

각 프로젝트 문서는 다음 중 5개 이상을 포함해야 한다.

1. 문제 정의
2. 책임 경계
3. 핵심 설계 결정
4. 버린 선택과 이유
5. 운영자가 확인해야 하는 흔적
6. 운영 검증
7. 역할과 책임
8. 설계 결론

## 톤 판단

첫 화면은 낮게 쓴다. 상세 문서는 깊게 쓴다. 직급과 숫자를 앞세우지 말고, 운영자가 실제로 확인할 수 있는 기록과 설계자가 버린 선택을 통해 전문성을 드러낸다.
